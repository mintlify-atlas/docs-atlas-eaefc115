Running `/ads test` produces a complete A/B test plan for your advertising campaigns. Claude builds a structured hypothesis, calculates the minimum sample size needed for statistical significance, estimates test duration based on your daily traffic, and writes platform-specific setup instructions for Meta Experiments, Google Experiments, LinkedIn A/B, or TikTok split tests. The goal is to run tests you can learn from — not tests that produce false confidence from insufficient data.

## Command

```bash
/ads test
```

Tell Claude what you want to test and on which platform. Provide your current conversion rate and daily click or impression volume so Claude can calculate sample size and duration.

**Example:**

```bash
/ads test

I want to test UGC video versus polished product video on Meta.
Current CVR: 2.8%, daily clicks: ~600, platform: Meta
```

<Note>
  `/ads test` is a quick planning utility. Claude does not generate a long-form report or append the community footer after this command.
</Note>

## Hypothesis framework

Every test Claude designs starts with a structured hypothesis in this format:

```
IF we [change / action]
THEN [metric] will [increase / decrease] by [estimated %]
BECAUSE [reasoning based on data or insight]
```

**Example hypothesis:**

```
IF we replace polished product shots with UGC creator videos
THEN Meta CTR will increase by 25–40%
BECAUSE Andromeda prioritizes diverse creative formats and UGC
consistently outperforms polished creative in 2025–2026 benchmarks
```

### Hypothesis quality checklist

Before finalizing the test design, Claude verifies:

- A single variable is being tested — isolate the change completely
- A specific metric is defined, not vague "performance"
- An estimated effect size is stated — needed for sample size calculation
- A timeframe is defined
- Success and failure criteria are clear before the test launches

## Statistical significance

Claude calculates required sample size using the standard formula:

```
n = (Z_alpha + Z_beta)² × 2 × p × (1 − p) / MDE²

Where:
  Z_alpha = 1.96  (95% confidence level)
  Z_beta  = 0.84  (80% statistical power)
  p       = baseline conversion rate
  MDE     = minimum detectable effect (relative %)
```

### Sample size lookup table

*Per variant. Assumes 95% confidence, 80% power.*

| Baseline CVR | 5% MDE | 10% MDE | 20% MDE | 30% MDE |
|-------------|--------|---------|---------|---------|
| 1% | 612,000 | 153,000 | 38,300 | 17,000 |
| 2% | 302,400 | 75,600 | 18,900 | 8,400 |
| 5% | 116,800 | 29,200 | 7,300 | 3,200 |
| 10% | 55,200 | 13,800 | 3,450 | 1,530 |
| 20% | 24,600 | 6,150 | 1,540 | 680 |

<Warning>
  If you're testing for a small effect (5% MDE) on a low-volume account, sample size requirements are very high. Claude will flag when a test is unlikely to reach statistical significance in a reasonable timeframe and recommend either accepting a larger MDE or running the test on a higher-traffic campaign.
</Warning>

## Test duration estimator

```
Duration = Required Sample Size / Daily Traffic per Variant

Minimum duration:    7 days  (capture weekly behavioral patterns)
Maximum recommended: 28 days (avoid seasonal drift distorting results)
```

Platform learning phase minimums:
- Google: 7–14 days
- Meta: 3–7 days
- LinkedIn: 7–14 days

### Duration quick estimates

| Daily clicks | 2% CVR, 20% MDE | 5% CVR, 20% MDE | 10% CVR, 20% MDE |
|-------------|-----------------|-----------------|-----------------|
| 100 | 189 days | 73 days | 35 days |
| 500 | 38 days | 15 days | 7 days |
| 1,000 | 19 days | 7 days | 4 days* |
| 5,000 | 4 days* | 2 days* | 1 day* |

*Minimum 7 days applies regardless of sample sufficiency — to capture weekly behavioral patterns.

## Platform-specific test setup

<Tabs>
  <Tab title="Meta Experiments">
    Use Ads Manager → Experiments tab. Do not manually duplicate ad sets — the Experiments tool ensures automatic audience splitting with zero overlap between variants.

    **Supported test types:** A/B (creative, audience, placement), Holdout, Brand Lift Survey.

    **Budget:** Split evenly across variants. Minimum $100/day per variant recommended for sufficient delivery speed.

    **Duration:** 7–14 days is typical. Meta auto-declares a winner at 95% confidence. Do not manually end the test before Meta signals completion.

    **Meta Incremental Attribution (April 2025):** AI-powered holdout testing now available for measuring true causal impact — what would not have happened without the ad. Available for accounts spending $5,000+/month.
  </Tab>
  <Tab title="Google Experiments">
    Create from an existing campaign → Campaign Experiments or use Ad Variations for copy-only tests.

    **Traffic split:** 50/50 recommended for the fastest results. Uneven splits require longer test duration.

    **Supported tests:** Bidding strategy, ad copy, landing page URL, audience layer.

    **Primary metric:** Define your primary metric (conversions, CPA, ROAS) before launch — Google uses this to declare a winner.

    **Duration:** 14–30 days recommended. Minimum two weeks for any bidding strategy test. Avoid making changes to the base campaign during the experiment period.
  </Tab>
  <Tab title="LinkedIn A/B">
    Duplicate the ad set in Campaign Manager with a single variable change. Set the same audience segment on both; LinkedIn rotates ads automatically.

    **Minimum budget:** $50/day per variant.

    **Key benchmarks:** CTR >0.44%, Lead Form CVR 13% (approximately 3.25× the typical landing page CVR).

    **Duration:** 14–21 days. LinkedIn's smaller daily impression volumes require longer test windows to reach statistical significance compared to Meta or Google.
  </Tab>
  <Tab title="TikTok Split Testing">
    Available in TikTok Ads Manager → Create A/B Test. Supported test types: targeting, bidding, creative.

    The platform auto-splits the audience to prevent contamination between variants.

    **Minimum duration:** 7 days. Recommended: 14 days.

    **Budget:** Minimum $20/day per ad group.

    **Creative tests:** Isolate the hook (first 2–3 seconds) as the primary variable — it's the single highest-leverage element in TikTok ad performance.

    **Smart+ Modular (2025):** TikTok's enhanced split testing supports modular test variables — you can lock targeting, creative, budget, or placement independently while automating the rest.
  </Tab>
</Tabs>

## What to test — priority order

Test the highest-impact variables first. Smaller changes should come only after you've exhausted the high-leverage opportunities.

<AccordionGroup>
  <Accordion title="High impact — test first">
    1. **Creative concept** — different messaging angles, not just color or font changes
    2. **Hook / first 3 seconds** — the video opening on Meta, TikTok, and YouTube
    3. **Offer structure** — pricing, discount type, free trial length
    4. **Landing page** — headline, CTA, form length
    5. **Bidding strategy** — Target CPA versus Target ROAS versus Maximize Conversions
  </Accordion>
  <Accordion title="Medium impact">
    6. **Audience targeting** — interest vs lookalike vs broad
    7. **Ad format** — static image vs video vs carousel
    8. **CTA button** — "Learn More" vs "Sign Up" vs "Shop Now"
    9. **Campaign structure** — CBO vs ABO, consolidated vs segmented ad sets
  </Accordion>
  <Accordion title="Low impact — test last">
    10. **Ad scheduling** — time of day, day of week
    11. **Device targeting** — mobile vs desktop
    12. **Minor copy variations** — word substitutions without a concept-level change
  </Accordion>
</AccordionGroup>

## Common testing mistakes

<AccordionGroup>
  <Accordion title="Testing too many variables at once">
    If you change the creative, the audience, and the landing page simultaneously, you can't attribute the result to any single change. Test one variable per experiment.
  </Accordion>
  <Accordion title="Ending tests too early">
    Checking results daily and pausing the losing variant before reaching statistical significance is the most common reason test results fail to hold up. Let the test run to completion.
  </Accordion>
  <Accordion title="Testing during atypical periods">
    Tests run during holidays, product launches, or platform algorithm changes produce results that don't generalize. Avoid running new tests during Black Friday week, major news events, or within two weeks of a platform policy change.
  </Accordion>
  <Accordion title="Ignoring the learning phase">
    On automated platforms like Meta and Google, edits during the learning phase reset the algorithm's optimization. Changing anything during an active test — including the control — voids the result.
  </Accordion>
  <Accordion title="Not documenting learnings">
    A test that can't be referenced six months later has half its value. Claude's test plan output includes a documentation template for recording what was tested, what was found, and how it influenced future campaigns.
  </Accordion>
  <Accordion title="Testing small changes when big changes are needed">
    If a campaign is performing at 20% of target CPA, you need a concept-level change — not a headline word swap. Optimize from a position of strength; innovate when performance is already failing.
  </Accordion>
</AccordionGroup>
